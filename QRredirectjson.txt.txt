1. If running locally (on your computer)
You use a file named .env located in the main folder of your project.

Create the file: If you don't have one, create a new file named exactly .env (no file extension).

Add your variables: Open that file in your code editor and type them out line-by-line. If running in production use NODE_ENV=production

Plaintext
NODE_ENV=development
MONGODB_URI=mongodb+srv://your_connection_string
SESSION_SECRET=super-secret-string-here
ADMIN_EMAIL=your@email.com
ADMIN_PASSWORD=your-password
ADMIN_PATH=my-secure-admin-panel
RECAPTCHA_SECRET_KEY=your_secret_key
RECAPTCHA_SITE_KEY=your_site_key
N8N_WEBHOOK_URL=your_webhook_url
Why it works: Your code uses a package called dotenv (imported at the very top of your server.js). That package automatically reads this file and loads those values into process.env.

2. If running on Render (Production)
You should not upload a .env file to your server. Instead, you use the web interface.

Log into your Render Dashboard.

Click on the name of your Web Service.

Click the Environment tab on the left sidebar.

Click Add Environment Variable.

Enter the Key (e.g., NODE_ENV) and the Value (e.g., production).

Save Changes. Render will automatically trigger a new deployment of your app to apply these new settings.

Best Practices to remember:
Never share your .env file: Do not push it to GitHub. Ensure .env is listed in your .gitignore file so it stays private.

Case Sensitivity: Environment variable names are traditionally uppercase (e.g., ADMIN_PATH).

No Quotes: You generally don't need quotes around the values in the .env file (e.g., ADMIN_PATH=my-admin is better than ADMIN_PATH="my-admin").

Restart after changes: Whenever you change an environment variable in your .env file, you must stop your node server (Ctrl+C) and start it again for the changes to take effect.
